SYMPTOM
-------
All five tests abort with:

  Instruction does not dominate all uses!
    %ptr.addr = getelementptr ptr, ptr %Ptrs, i32 %i
    ...
  LLVM ERROR: Broken module found, compilation aborted!

The IR verifier detects that cloned instructions are inserted at a point
that does not dominate their uses (or that uses of loop-local values appear
outside the loop). opt then aborts (exit code -6, SIGABRT).

ROOT CAUSE
----------
In TPlanLowering_lower() (TPlanLowering.cpp), the IRBuilder is positioned
at the function entry block:

  Builder.SetInsertPoint(&F.getEntryBlock().front());

Then TPRegionBlock::execute() and TPBasicBlock::execute() walk the block
graph in construction order.  Only TPIRBasicBlock::execute() repositions
the builder (to the first non-PHI of its associated IR basic block).  All
other TPBasicBlock::execute() calls (tensor.latch, tensor.ph, tensor.body,
middle.block, scalar.ph) emit their cloned instructions wherever the builder
currently is.

Because loop-body values (%i, %k, GEP results, load results, etc.) are
defined inside loop blocks, cloning them at the entry block (or at whatever
arbitrary builder position is current) violates SSA dominance: a use
appears before its definition in CFG post-order.

AFFECTED RECIPE EXECUTE() IMPLS
--------------------------------
  TPWidenGEPRecipe::execute()   -> Builder.Insert(GEPInst->clone())
  TPWidenLoadRecipe::execute()  -> Builder.Insert(LoadInst->clone())
  TPWidenStoreRecipe::execute() -> Builder.Insert(StoreInst->clone())
  TPWidenRecipe::execute()      -> Builder.Insert(Inst->clone())
  TPWidenCastRecipe::execute()  -> Builder.Insert(CastInst->clone())

All of these blindly clone the original instruction and insert at the
current builder position without remapping operands or repositioning the
builder to the instruction's original dominating block.
